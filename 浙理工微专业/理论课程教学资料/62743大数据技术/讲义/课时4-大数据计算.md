## 4.2 Apache Spark：批处理的计算革命

### 4.2.1 核心定义与地位

**Apache Spark** 是目前全球应用最广泛的**分布式统一计算引擎**。它的诞生是为了解决早期MapReduce（Hadoop）反复读写硬盘导致的性能瓶颈。Spark 不仅是批处理之王，还试图通过微批次（Micro-batch）的方式统一批处理和流处理。

### 4.2.2 核心机制：内存计算（In-Memory Computing）

- **原理**：传统引擎每执行一步都要把数据写入硬盘，再读出。Spark将中间计算结果直接存储在**内存（RAM）**中，减少了90%以上的磁盘I/O。
- **类比**：就像你在做数学题，传统方式是算一步就把草稿纸收进抽屉（硬盘），下次用时再拿出来；Spark是把所有草稿铺在桌面上（内存），随用随取。
- **优势**：速度比传统MapReduce快10~100倍，特别适合需要多次迭代的机器学习算法。

### 4.2.3 核心抽象：RDD与DAG

- **RDD（弹性分布式数据集）**：Spark的基石。它是一个不可变的、可分区的、里面装满了数据的集合。你可以把它想象成一个巨大的、分布在成千上万台机器上的数组。
- **DAG（有向无环图）**：Spark会将你的计算逻辑（比如"过滤->映射->聚合"）构建成一个图。它会自动优化这个图的执行路径，避免不必要的计算。

### 4.2.4 生态栈（Unified Stack）

Spark之所以流行，是因为它"啥都能干"：
- **Spark Core**：基础计算引擎。
- **Spark SQL**：处理结构化数据（替代Hive）。
- **Spark Streaming**：通过微批次处理实时数据（准实时）。
- **MLlib**：分布式机器学习库。
- **GraphX**：图计算库。

### 4.2.5 适用场景

- **离线数仓建设**（ETL）：每天凌晨清洗和处理TB级历史数据。
- **复杂数据分析**：需要Join多张大表进行关联分析。
- **机器学习训练**：需要反复迭代计算模型的场景。

## 4.3 Apache Flink：流处理的事实标准

### 4.3.1 核心定义与地位

**Apache Flink** 是**第三代流处理引擎**的代表。如果说Spark是"把批处理做到极致"，Flink则是"把流处理做到极致"。它认为**一切数据都是流**，批处理只是流的一个特例（有界流）。

### 4.3.2 核心机制：事件驱动与状态计算

- **事件驱动（Event-Driven）**：Flink是真正的"来一条数据处理一条"，而不是像Spark那样"攒一批再处理"。
- **状态管理（State Management）**：这是Flink最强大的地方。在处理数据时（比如计算累计销售额），Flink会在内存或RocksDB中维护一个"状态"（比如当前总和是多少）。即使程序重启，这个状态也不会丢。
- **时间语义（Time Semantics）**：
    - *Event Time（事件时间）*：数据本身自带的时间戳（如用户下单时间）。这是最准确的，也是Flink的强项。
    - *Processing Time（处理时间）*：数据到达计算引擎的时间。

### 4.3.3 容错机制：Chandy-Lamport 算法

Flink能够在节点故障时，精确地恢复到故障前的状态，保证数据**只被处理一次（Exactly-Once）**。这对于金融交易至关重要。

### 4.3.4 适用场景

- **实时风控**：检测到异常交易瞬间拦截。
- **实时大屏**：双十一成交额、抖音在线人数实时跳动。
- **金融对账**：银行流水实时匹配。

---

## 4.4 Spark vs Flink：技术特性对比

| 维度 | Apache Spark | Apache Flink |
| :--- | :--- | :--- |
| **计算模型** | **微批次（Micro-Batch）** | **纯流（Native Streaming）** |
| **延迟** | 秒级 ~ 分钟级 | **毫秒级** |
| **状态管理** | 较弱（主要依赖外部存储） | **极强（内置状态后端，支持超大状态）** |
| **时间语义** | 侧重处理时间，事件时间支持一般 | **完美支持事件时间，处理乱序数据能力强** |
| **吞吐量** | **极高**（得益于内存计算优化） | 高（稍逊于Spark，但足以满足绝大多数场景） |
| **成熟度** | 极高，生态完善 | 高，流处理领域事实标准 |

## 4.5 架构思想的博弈：Lambda vs Kappa

当我们决定用Spark还是Flink后，还需要决定**系统的整体架构**。

### 4.5.1 Lambda 架构（双车道）

- **设计初衷**：为了解决"既要快又要准"的矛盾。
- **架构组成**：
    1.  **Speed Layer（快车道）**：用Flink处理实时流，结果可能有误差（因为只看了最近的数据）。
    2.  **Batch Layer（慢车道）**：用Spark处理全量历史数据，结果准确但慢。
    3.  **Serving Layer**：将快慢结果合并展示。
- **痛点**：维护两套代码（一套实时、一套离线），开发成本极高，容易不一致。

### 4.5.2 Kappa 架构（单车道）

- **设计初衷**：受限于Lambda的复杂性，LinkedIn的Jay Kreps提出了Kappa架构。
- **核心理念**：**去掉批处理层，只用一条流处理管道**。
    - 所有的数据都以流的形式进入消息队列（如Kafka）。
    - 如果需要重新计算历史数据，只需重置Kafka的消费偏移量（Offset），让流处理引擎重新读一遍历史数据即可。
- **优势**：架构极其简洁，只需维护一套代码。
- **挑战**：对消息队列（存储历史数据的能力）和流计算引擎的稳定性要求极高。

## 4.6 现代资源调度：Kubernetes (K8s) 的统治

无论使用Spark还是Flink，现代大数据平台都不直接把任务跑在物理机上，而是跑在容器编排系统 **Kubernetes** 上。
- **弹性伸缩**：流量高峰时自动增加计算节点，低谷时释放，极大降低成本。
- **存算分离**：计算资源和存储资源独立调度，互不影响。

---

## 4.7 如何选择你的武器？

- **如果你主要做报表、数据分析、机器学习** -> 选 **Spark**（性价比高，生态好）。
- **如果你主要做实时期限、金融交易、监控预警** -> 选 **Flink**（低延迟，状态强）。
- **如果你的团队很小，不想维护两套系统** -> 选 **Kappa 架构**（Flink + Kafka）。
- **如果你需要极高的准确性，且不怕维护成本** -> 选 **Lambda 架构**。
